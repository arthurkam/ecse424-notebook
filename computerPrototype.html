Design Evolution

After going through the user testing process and gathering as much feedback as possible, multiple potential design revisions were discovered. However not only did the test subjects discover minor design flaws, but other members of the NavSense team did as well.

When the test subjects were first exposed to the low-fidelity paper UI, it was pointed out that the reference point editing screen was too cluttered. This caused a slight issue when users were trying to simply set their vibration pattern and name their reference. To correct this issue, the edit page was changed to be more intuitive, prompting the user the first enter the name of their reference point at the top of the screen (instead of towards the bottom), followed by the notification pattern. Additionally, at the bottom of the page a small map view will display the location of the reference point to remind the user of the exact location.

The second important issue that was brought up was the fact that users didn’t really want to create their own vibration pattern from scratch. Whenever the custom vibration pattern section was encountered, the users would simply ask to use a pre-set one instead. For this reason, the NavSense team decided to include only pre-defined vibration patterns for the users to choose from. Since the task at hand does not actually require the need for overly customizable vibration patterns, we realized that there was no real advantage of including this feature. Moreover, this change will allow us to inhibit users from creating overly complicated patterns which may reduce the usefulness of the application in the long run.

The third and most important issue that was rectified on our high fidelity prototype was how the vibrations were transmitted to the user. One of the main things that we noticed with the vibrations on the Pebble was that they were not always distinguishable from one another. It was easy for the users to determine the difference between two and three vibrations, but once different vibration lengths are employed it might get more complicated. Although these more complex vibration patterns were not able to be tested on our low-fidelity prototype, all of the members of the NavSense Team agree that this is a design issue that needs to be changed. In order to make the vibration patterns more distinguishable, an additional set of vibrations will be added to the beginning of an instruction. This way when a user senses feedback indicating the location of one of their reference points, they will be ready to start paying attention to the pattern. If this change wasn’t employed, the user might not necessarily feel the beginning of the pattern and may possibly confuse it for a different reference point.

Usability Test Procedure, Population and Goals

The following document presents the detailed usability goals and test procedure. Examiners are encouraged to attentively read this document and apply its content to the best of their knowledge.

How many examiners are required ?

Ideally, 2 or 3 evaluators are required to conduct the usability testing of the prototype. This number ensures that enough people are there to observe the user in case one examiner did not see a particular behaviour. Since this experiment contains quite a lot of movements and displacement, being more numerous helps in carrying the required user guide, data entry sheet, etc.

Distinct tasks, to which example names have been giver, should be assigned to each examiner and a described below :

1. The mother : This examiner is the one responsible to conduct the user introduction, explanation of the projected experiments and each evaluation task explanation. He is also the one in charge to carry the test material throughout the experiment and make sure that the subject is not exposed to situational dangers (i.e. car, construction site, horse, etc.). The mother should still be observing the subject throughout the experiment.

2. The eye : This examiner is there to do a detailed observation and to quantify the elements presented in the provided data entry sheets. If testing is done with more than 2 examiners, multiple examiners take this role.

What equipment will the examiners need ?

The following list contains the material required in order to conduct the usability testing.

Hardware :

- Android smartphone or tablet (as long as it has bluetooth, a GPS and wifi/data connection)

- Pebble Smartwatch

- A second smartphone with a compass app of your choice and/or a physical compass (and knowing how to read it)

Documents

- 1 x User guide

- 1 x Usability test procedure

- N x Pre-test questionnaire

- N x Post-test questionnaire

- M x Data entry sheet

Where N is dependant on the number of testers. Pre-test and post-test questionnaire should be done for every tester. M is dependant on the number of testers and examiners. Each tester should have its data entered at least one in the data entry sheet during the testing. However, having more than one data entry sheet per tester gives more error tolerance to the test by double validating every entry. In this line of idea, having 2 data entry sheet per tester is advised.

How should your prototype be handled ?

Note that since the pebble smartwatch are provided by the NavSense team, proper care of this item should be taken. Never leave it unattended or in untrustworthy hands. If you do not have an Android device at your disposal, a member of the NavSense team can provide you with one but may ask to be present during the testing. One of your team members will be the official responsible for the borrowed material. In the event that you lose any of the provided material (pebble, android device, etc.) he will be responsible for coordinating the replacement of the lost item.

The provided user guide contains all the required information to install and operate the android and the smartwatch application. It also contains usage examples placed in context and troubleshooting advice.

How will you instruct your examiners to proceed ?

It is important that the examiners do not interfere with or affect the actions of the user during the test procedure. They should simply give the test user all of the required documentation along with the confidence to succeed in this test session. It is also critical that the examiners are thorough in their observation of the user's movements so that they can gather as much informative feedback from the test as possible. It is expected that the user will run into some issues during the test considering that they have never used NavSense before. Nonetheless, the examiners must allow the test users to figure out the functionality of the app on their own. It is acceptable however for an examiner to help a test subject if they are about to encounter a hazardous situation during the course of the navigation. To ensure that the test data is consistent across all test subjects, we are requesting that the tests for NavSense take place at the Y-intersection on the McGill campus.

How should the examiners treat the test subjects, what should they tell them ?

A brief observer introduction is presented in a separate document named “Observer Briefing Document”. It contains an outline of the elements to take care of during the experimentation.

A brief user introduction document is presented in a separate document named “User Briefing Document”. It contains a brief explanation of the project that the test subject should know and general instructions for the experimentation. The examiners should let the user read the document and answer any question that he or she might have concerning the test procedure.

What should the examiners avoid doing?

The examiners should definitely avoid telling the test user what to do in the middle of a task. It is important they do not in any way affect the outcome of a test by communicating any additional information to the user. In the event that the user is completely stuck in the experiment and asks for help, you may assist him.

What should the examiners avoid telling the test subjects?

As previously mentioned, the examiners should avoid telling the users what to do in order to avoid influencing their actions. They should also avoid telling the users if they are incorrectly using the NavSense application so that the entire natural process can be observed.

What/How/When should the examiners measure ?

The examiners should first ensure that the user fills out the pre and post-test questionnaires. This will allow them to gain a rough idea of the type of user they are dealing with in terms of technical experience and navigation.

Once the pre-test questionnaire is filled out, the actual usability test will begin. The examiners should fill out the cells in the data collection sheet associated with each task by measuring the time required to complete them. For 2 tasks, examiners are asked to estimate to the best of their abilities the orientation error in degrees between the user’s claimed direction and the actual direction.

During all of the tasks, users are asked to think aloud, saying when they felt a vibration pattern. Every time a vibration pattern is felt, the examiner should place a mark/check/cross in the appropriate box at the bottom of the data collection sheet. When the user did not report a vibration but the examiner KNOWS that there should have been one, the examiner should place a mark/check/cross in the appropriate box at the bottom of the data collection sheet. The same concept applies to the different vibration patterns: if it is correctly or incorrectly identified, place a mark/check/cross in the appropriate box.

User Population:

The user population for this test should ideally be composed of students on/around the McGill University campus. Since the test procedures below define reference points around campus, they will be most effective if performed within the area that the points define. Ideally the test users will be somewhat unfamiliar with the area and have a poor sense of orientation.

Usability Goals:

1. User should be able to figure out how to add a single reference point on the map within 30 seconds of opening the application.

2. User is able to customize or delete the information corresponding to the reference point within 45 seconds.

3. User can feel the vibration patterns 75% of the time.

4. User is able to successfully identify the vibration pattern corresponding to their reference point 85% of the time.

5. User finds his reference point and is able to properly orient himself with respect to his reference using the haptic feedback within a 30° error and 30 seconds
